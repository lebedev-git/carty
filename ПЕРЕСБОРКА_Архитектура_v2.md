# Пересборка «Создание карт» → Фабрика колод (целевая архитектура v2)

> Цель документа: зафиксировать **логику и архитектуру** обобщённой системы до выбора стека.
> Опирается на разбор текущего n8n-процесса (`Создание_карт_АРХИТЕКТУРА.md`) и полные промпты (`_promts_bazovye.txt`).
> Статус: черновик для согласования. Стек — отдельный, второй шаг.

---

## 1. Какую задачу мы решаем (переформулировка)

**Продукт:** колода учебных карт для обучения работе с ИИ. Колода ≈ 72 карты: **36 «Я»** (индивидуальные) + **36 «МЫ»** (командные). Каждая карта — реалистичный управленческий мини-кейс (Описание + Задание), бьющий в реальную «боль» аудитории, с собственным изображением и (опц.) доп. материалом.

**Что есть сейчас:** система жёстко настроена под одну аудиторию («Московское долголетие»). Всё ядро — матрица поиска, лучи «ЗВЕЗДА», стоп-листы, цветовые гаммы — **захардкожено в ячейках Google Sheets**. Сменить аудиторию = вручную переписать ядро.

**Что хотим:** я ввожу тему — например «колода для школ» — и дальше система **сама**:
1. понимает, что такое организация этого типа (структура, стейкхолдеры, реальные проблемы, контекст);
2. строит под неё **ядро** (матрицу тем + лучи + стоп-листы + визуальную айдентику);
3. ищет реальные кейсы;
4. отбирает и оценивает их;
5. собирает карты «Я» и «МЫ»;
6. генерирует промпты изображений под типы;
7. финально собирает карты (далее — рендер в Figma).

**Ключевой сдвиг:** ядро колоды перестаёт быть константой и становится **генерируемым артефактом**. Это одновременно и механизм обобщения, и главный рычаг качества.

## 2. Центральная идея: «Профиль колоды» (Deck Profile)

Вводим главный артефакт — **Профиль колоды**: структурированное (не текст, а данные) описание всего, что нужно для генерации конкретной колоды. Он создаётся один раз на старте, **проверяется/правится методологом**, замораживается — и дальше все стадии читают его.

```
DeckProfile {
  audience:            { name, description, segments[] }      // "Школы РФ", портреты
  functional_blocks:   [{ id:M1, name, focus }]               // 4–6, генерируются под домен
  stakeholder_clusters:[{ id:C1, name, question }]            // 4–6, генерируются
  matrix:              [{ m, c, seed_themes[] }]               // M×C — карта поиска кейсов
  zvezda_rays:         [{ name, patterns:[6×{name,task_type}]}]// ось «МЫ» (адаптируется)
  content_types:       [TEXT,AUDIO,IMAGE,VIDEO,INTERACTIVE,DATA] // ось «Я» (стабильна)
  palettes:            { by_type:{...}, by_ray:{...} }         // визуальная айдентика
  taboo_list:          [...]                                   // этический стоп-лист
  tone_rules, term_rules                                       // тон, запрет англицизмов и т.п.
  constraints:         { name_len, desc_len, task_len, ... }   // размерные лимиты
  quality_targets:     { min_cards_per_axis, min_score }       // напр. 16/16, score ≥80
}
```

Почему это правильно:
- **Обобщение** — сменить аудиторию = сгенерировать новый Профиль, код не трогаем.
- **Качество** — если ядро глубокое и выверенное, всё ниже по потоку резко лучше. Это точка приложения усилий №1.
- **Контроль** — методолог правит структуру (матрицу, лучи, стоп-листы) до массовой генерации, а не вычитывает 72 готовые карты.

## 3. Целевой конвейер (стадии)

Каждая стадия = вход → что делает **LLM** vs что делает **код** → выход + контроль качества.

### Стадия 0 — Профайлер домена (НОВОЕ)
- **Вход:** бриф темы («колода для школ», опц. уточнения).
- **LLM + веб-поиск:** исследует тип организации — структура, стейкхолдеры, типовые проблемы, отраслевой контекст, терминология. Строит черновик `DeckProfile` (матрица M×C, лучи, стоп-листы, гаммы).
- **Код:** валидирует полноту (нет пустых ячеек, лучей ровно 6, гаммы заданы).
- **👤 Чекпоинт:** методолог ревизует/правит Профиль → «заморозка».
- **Выход:** утверждённый `DeckProfile`.

### Стадия 1 — Источник кейсов
- **Код:** из `matrix` генерирует задания на поиск (≥2 покрытия каждого блока M, разные кластеры C, без дублей ячеек M×C).
- **LLM:** формирует поисковые промпты по шаблону (роль аналитика, гео, период, стоп-лист).
- **🔌 Веб-поиск (ДОСТРОЙКА — сейчас этого шага нет):** реально ищет 4–5 кейсов на промпт.
- **LLM-экстрактор:** превращает найденное в структурированные `Case{название, суть, ссылка, тип_проблемы, M×C}`.
- **Выход:** пул сырых кейсов.

### Стадия 2 — Валидация и скоринг
- **LLM-аудитор:** дедупликация → скоринг 0–100 по 5 критериям → типизация «Я» (Инструментальная) / «МЫ» (Стратегическая).
- **Код:** считает баллы по критериям как отдельные поля, проверяет квоты (≥16 «Я» 80+, ≥16 «МЫ» 80+); если не добрали — триггерит дозапрос кейсов (возврат к Стадии 1).
- **👤 Чекпоинт (опц.):** методолог утверждает пул кейсов.
- **Выход:** валидированные кейсы с оценкой и группой.

### Стадия 3 — Генерация карт
- **Код-планировщик:** решает комбинаторику **детерминированно** — 36 карт = 6 осей × 6 паттернов, паттерн ровно 1 раз, кейс ≤3 раз/ось. Подбор «паттерн↔кейс» по смысловой близости (эмбеддинги).
- **LLM (по одной карте):** на вход (кейс + паттерн + профиль боли аудитории + few-shot) → одна карта `{Название, ТИП/Луч, Description, Task, Доп_материал}`.
- **Контроль качества (2 гейта, см. раздел 5):** валидатор ограничений + «тест боли» (LLM-judge) → слабые карты перегенерируются точечно.
- **Выход:** `Карты Я` + `Карты МЫ`.

### Стадия 4 — Визуальные промпты
- **Код:** собирает каркас изображения из Профиля — фон-константа + гамма по типу/лучу (это шаблон, **не нужен LLM**).
- **LLM:** генерирует только переменные части — `[центральный образ]` и `[настроение]` под смысл карты.
- **Выход:** промпт изображения на карту.

### Стадия 5 — Сборка и рендер
- **Код:** склейка карты (текст + изображение + доп. материал) в единый объект.
- **🔌 Рендер (будущее):** генерация самих изображений по промптам → **Figma** (через Figma API/плагин) формирует визуальные карты по шаблону.
- **Выход:** готовая колода.

### Схема потока
```
Бриф темы
  └─0─ Профайлер(LLM+поиск) ──👤──▶ DeckProfile (заморожен)
            │
  ┌─1─ Поисковые промпты(код) → Веб-поиск(🔌) → Экстрактор(LLM) ─▶ Сырые кейсы
  │         ▲                                                          │
  │         └────────── дозапрос если не добрали квоту ◀──────────┐    │
  └─2─ Аудитор(LLM) + Квоты(код) ──👤──▶ Валидированные кейсы ────┘────┘
            │
  └─3─ Планировщик(код: 6×6, ≤3) + Генератор карт(LLM по 1) + Гейты(валидатор+judge) ─▶ Карты Я/МЫ
            │
  └─4─ Каркас(код: фон+гамма) + Образ/настроение(LLM) ─▶ Промпты изображений
            │
  └─5─ Сборка(код) + Рендер(🔌 Figma) ─▶ Колода
```

## 4. Принципы пересборки (почему так лучше текущего)

1. **Разделить три слоя:** данные (Профиль) ≠ промпты (версионируемые шаблоны) ≠ оркестрация (код). Сейчас всё смешано в ячейках Sheets.
2. **Детерминизм там, где можно.** Комбинаторику (6×6, квоты 16/16, ≤3), подстановку гамм, проверку размеров — делает **код**, а не LLM. LLM отвечает только за творческую часть. Это убирает «хирургические» парсеры битого JSON.
3. **Structured output + схема + ретрай** вместо регэкспов и `JSON.parse(...).lastIndexOf('}')`. Невалидный ответ → автоматический повтор, а не падение.
4. **Генерация по одному элементу**, а не «36 карт одним JSON». Богаче контекст на карту, дешевле переген одной слабой карты.
5. **Качество как явные гейты** (раздел 5), а не «надежда на промпт».
6. **Человек в контуре** на 1–2 чекпоинтах (Профиль, пул кейсов). Методолог управляет рычагами, а не вычитывает результат.
7. **Состояние и идемпотентность.** Колода = объект со статусами (`brief → profiled → sourced → validated → cards → visuals → assembled`). Можно возобновить, перегенерировать только часть.

## 5. Контроль качества (главная боль заказчика)

Три уровня гейтов, каждый — с авто-возвратом на переген:

**Гейт A — качество кейса (Стадия 2).** Скоринг 0–100 по 5 критериям становится **структурным** (каждый балл — поле), порог 80+, квоты 16/16 проверяет код. Не добрали — дозапрос.

**Гейт B — «тест боли» карты (Стадия 3).** Формализуем существующий «тест релевантности» в отдельного **LLM-judge** со шкалой: узнаваемость («это про нас!»), наличие конфликта/дефицита, реалистичность, соответствие паттерну. Карта ниже порога → переген с уточнённым промптом.

**Гейт C — валидатор ограничений (Стадия 3, код).** Жёсткие правила проверяются программно:
- размеры: Название 15–20, Description 300–350, Task 180–220 симв.;
- запреты: конкретные даты, упоминания «ИИ/нейросеть», англицизмы (KPI/CRM/ROI…), кавычки `"`;
- тип/луч строго из списка; «Доп. материал» ∈ {Генерация, Трансформация};
- для изображения — гамма соответствует типу/лучу.
Нарушение → **точечный фикс** конкретного поля («сократи Description до 350»), а не полный переген.

**Рычаги глубины контекста (то, что просил заказчик):**
- Стадия 0 и 1 **заземляют** колоду на реальности через веб-поиск, а не на «фантазии модели».
- Подбор «паттерн↔кейс» по эмбеддингам → карты точнее ложатся на смысл.
- Few-shot из лучших карт + явный «профиль боли» аудитории в промпте генерации.

## 7. Куда встаёт Figma и медиа-рендер

- **Стадия 4** уже даёт структурированный промпт изображения (фон+гамма+образ+настроение) — это контракт для рендера.
- **Рендер изображений:** отдельный сервис (image-API) по промптам → файлы/URL.
- **Figma:** на Стадии 5 — шаблон карты в Figma + Figma API/плагин, который раскладывает поля (Название/Описание/Задание/изображение/гамма) по макету. Профиль уже содержит визуальную айдентику (гаммы), что упрощает шаблонизацию.
- Это **аддитивный** слой: ядро (стадии 0–4) от него не зависит, поэтому MVP можно собрать без Figma и подключить её позже.

## 8. Открытые развилки (согласуем перед MVP)

Ниже — решения, влияющие на форму системы. Мои рекомендации помечу при обсуждении:

1. **Степень автоматизации Профиля (Стадия 0)** — авто + ревизия человеком, или полностью авто, или человек задаёт матрицу руками.
2. **Источник кейсов** — реальный веб-поиск/грунтинг, или LLM-генерация правдоподобных, или гибрид.
3. **Каркас «Я»/«МЫ»** — фиксированный (адаптируем примеры) или генерируется целиком под аудиторию.
4. **Объём первого MVP** — докуда доводим сквозной прототип (до карт-текста / до промптов изображений / до Figma).

После согласования этих 4 пунктов — фиксирую целевую логику, и переходим к **шагу 2 (стек)** и затем к прототипу MVP.

## 9. Согласованные решения (зафиксировано)

| # | Развилка | Решение |
|---|---|---|
| 1 | Профиль (ядро) | **Авто-генерация + обязательная ревизия человеком** перед массовой генерацией. Стадия 0 строит черновик, методолог правит и замораживает. |
| 2 | Источник кейсов | **Deep research через LLM по API** — не одиночный веб-поиск, а многошаговое исследование (агент-исследователь: серия запросов → чтение источников → извлечение и верификация кейсов). Сильнейшее заземление на реальности. |
| 3 | Каркас «Я»/«МЫ» | **Фиксированный** — 6 типов контента и 6 лучей ЗВЕЗДА стабильны (модель ИИ-компетенций). Под аудиторию адаптируются только примеры, тон, гаммы. |
| 4 | Объём MVP | **До карт-текста:** Тема → Профиль → кейсы → готовые карты «Я»/«МЫ» (текст) с гейтами качества. Без изображений и Figma (добавим позже). |

### Следствия для архитектуры
- **Каркас → конфиг-константа в коде** (не генерируется): 6 типов «Я», 6 лучей «МЫ», их 36+36 паттернов с типом задачи (Генерация/Трансформация), базовые гаммы. Профиль наследует каркас и **дописывает** к нему доменную часть (матрица M×C, примеры, тон, стоп-листы).
- **Стадия 1 = deep-research модуль:** агент-исследователь по каждой ячейке матрицы M×C ведёт мини-исследование (несколько поисков + чтение + извлечение `Case` со ссылкой-источником), затем дедуп/верификация. Это заменяет «выдуманные кейсы» текущей системы.
- **Два чекпоинта человека в MVP:** (1) ревизия Профиля; (2) опц. ревизия пула валидированных кейсов перед генерацией карт.

## 10. Контур MVP (что именно собираем первым)

**Вход:** бриф темы (напр. «колода для школ» + 1–2 строки уточнений).
**Выход:** ~72 карты-текста (36 «Я» + 36 «МЫ»), прошедшие гейты качества, в просматриваемом/правимом виде.

**Стадии в составе MVP:**
1. **Профайлер (0):** deep-research по типу организации → черновик `DeckProfile` (матрица M×C, примеры под лучи/типы, стоп-листы, тон). → **👤 ревизия** → заморозка.
2. **Источник кейсов (1):** по матрице — deep-research агент → структурированные `Case` с источниками → дедуп.
3. **Валидация (2):** скоринг 0–100 (структурный, по 5 критериям) + типизация «Я»/«МЫ» + контроль квот (≥16/≥16, 80+) кодом; дозапрос при нехватке. → опц. **👤 ревизия**.
4. **Генерация карт (3):** планировщик-комбинаторика (6×6, кейс ≤3×, подбор паттерн↔кейс) + генерация карты **по одной** + гейты: валидатор ограничений (размеры/запреты) + LLM-judge «тест боли» + точечный переген.

**Вне MVP (следующие итерации):** промпты изображений (Стадия 4), рендер медиа, Figma (Стадия 5), полноценный UI (в MVP допустимы Sheets/CLI/простой веб как интерфейс).

**Критерий готовности MVP:** на вход «колода для школ» получаем валидную колоду карт-текста, где ≥90% карт проходят гейты автоматически, матрица покрыта без дублей, квоты 16/16 соблюдены.

### Что осталось решить перед стеком (шаг 2)
- Где живёт LLM-вызов: **Claude API напрямую** vs через OpenRouter (как сейчас) — влияет на structured output/инструменты deep-research.
- Нужен ли в MVP веб-интерфейс или достаточно запуска по команде + просмотр результата в таблице.
- Бюджет/лимиты на deep-research (глубина = качество vs стоимость/время).
