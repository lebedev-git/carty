## 1. Назначение системы (что это вообще делает)

Это **конвейер генерации контента для учебного тренажёра по работе с ИИ**. Конечный продукт — «колоды карт»: наборы учебных кейсов-заданий, по которым обучаемые тренируются решать управленческие задачи с помощью ИИ.

Целевые аудитории (из промптов): управленческая команда проекта «Московское долголетие», а также проект «МАЯК Артек».

Каждая **карта** — это учебная ситуация со следующими полями:
- `Номер` (порядковый номер карты),
- `Исходный кейс` (ссылка на номер кейса-первоисточника),
- `Название`,
- `Тип` / `Луч` (компетентностная ось: напр. «Знание и навыки», «Трансформация», «Генерация»),
- `Описание` (Description) — фабула ситуации,
- `Задание` (Task) — что должен сделать обучаемый,
- `Допматериал` — вспомогательные входные материалы (справки, таблицы, сценарии).

Карты делятся на два класса:
- **«Я»** — индивидуальные карты,
- **«МЫ»** — командные/коллективные карты.

Система устроена как **набор из 6 независимых конвейеров**, каждый запускается своим триггером по Google Sheets (опрос раз в минуту). Это НЕ один сквозной поток — это 6 параллельных автоматизаций, живущих в одном workflow и связанных общими таблицами.

## 3. Карта связей (высокоуровнево)

Шесть веток, по триггерам:

```
[T2] Базовые промты ─→ AI Agent ─→ Code ─→ Append «Промты для поиска»        (П1: поисковые промпты)

[T1] Базовые промты ─→ Get «Кейсы(валидр.)» ─→ Aggregate ─→ AI Agent1 ─→ Code2 ─→ Update «Кейсы(валидр.)»   (П2: валидация/аналитика кейсов)

[T3] Базовые промты ─→ Get «Кейсы(валидр.)(копия)» ─→ If(ТИП=="Я")
        ├─ TRUE  ─→ [РАЗОРВАНО] … Aggregate1 ─→ Code8 ─→ AI Agent2 ─→ Code3 ─→ Append/Update «Карты Я»
        └─ FALSE ─→ Aggregate2 ─→ Code7 ─→ AI Agent3 ─→ Code4 ─→ Append/Update «Карты МЫ»                    (П3: генерация карт)

[T ] МАЯК ТЕСТ/Базовые ─→ Get «Карты Я» ─→ Aggregate3 ─→ AI Agent4 ─→ Code1 ─→ Append/Update «Карты Я»       (П4: медиа-промпты для карт Я)

[T4] МАЯК ТЕСТ/Базовые ─→ Get «Карты МЫ» ─→ Aggregate4 ─→ AI Agent5 ─→ Code5 ─→ Append/Update «Карты МЫ»      (П5: медиа-промпты для карт МЫ)

[T5] Законодат.власть/Лист9 ─→ Loop ─→ Wait ─→ AI Agent6 ─→ Create Google Doc ─→ Markdown→HTML ─→ Code6 ─→ HTTP(Docs batchUpdate)  (П6: доп. материалы в Google Docs)

[orphan] Edit Fields (хардкод-данные кейсов) ─→ Code9 (тупик, никуда не пишет)   (тестовый фрагмент)
```

Используемая ИИ-модель везде — через **OpenRouter**, семейство Google Gemini 3 (см. раздел 8).

## 4. Детальный разбор конвейеров

### Конвейер 1 — Генерация базовых поисковых промптов
**Назначение:** из конфигурации колоды получить список «поисковых промптов» (для последующего наполнения кейсов реальным контекстом).

1. **Google Sheets Trigger2** — лист `Базовые промты` (МАЯК Артек), опрос `everyMinute`. Срабатывает на новую/изменённую строку конфигурации.
2. **AI Agent** (`promptType: define`):
   - *User text:* `{{ $json["Создание Базовых промтов"] }}` — берётся из ячейки.
   - *System message:* конкатенация `Тема колоды` + `Аудитория` + `Ядро` + `Инструкция`.
   - Модель: `google/gemini-3.1-pro-preview`, `responseFormat: json_object`.
   - Ожидаемый выход: JSON-массив строк-промптов.
3. **Code in JavaScript** — парсит `output` (строку или объект) в массив, мапит в `{ номер: i+1, промпт }`.
4. **Append row in sheet1** → лист `Промты для поиска`, столбцы `Number`, `Prompt`. (Операция `append` — всегда дописывает.)

### Конвейер 2 — Валидация и аналитика кейсов
**Назначение:** прогнать имеющиеся кейсы через ИИ-аналитику, проставить оценку, аналитику и тип проблемы.

1. **Google Sheets Trigger1** — лист `Базовые промты` (триггер по конфигу).
2. **Get row(s) in sheet5** — читает все строки листа `Кейсы(валидрованные)`.
3. **Aggregate** — собирает все строки в один массив `data`, **исключая** поля `Оценка, Аналитика, ТИП, row_number, Ссылка, Статус, col_10, Тип проблемы` (т.е. подаёт агенту «сырьё» без прошлых оценок).
4. **AI Agent1**:
   - *User text:* `JSON.stringify($json.data)`.
   - *System message:* `Тема колоды` + `Аудитория` + **`Валидация`** + `Инструкция` (берутся из строки Trigger1).
   - Модель: `google/gemini-3-flash-preview`, json_object.
5. **Code in JavaScript2** — устойчивый парсинг с «хирургией» битого JSON (обрезает до последней `}` и закрывает массив); мапит каждый элемент через `getValue()` по списку возможных ключей в `{ номер, аналитика, оценка, группа }`.
6. **Update row in sheet** → `Кейсы(валидрованные)`, **матчинг по `Номер`**, обновляет `Аналитика`, `Оценка`, `ТИП` (`Тип проблемы` затирается пустым `=`).

### Конвейер 3 — Генерация карт «Я» и «МЫ» (ядро системы)
**Назначение:** из валидированных кейсов сгенерировать готовые учебные карты двух классов.

1. **Google Sheets Trigger3** — лист `Базовые промты`.
2. **Get row(s) in sheet1** — читает лист `Кейсы(валидрованные) (копия)`.
3. **If** — условие `{{ $json['ТИП'] }} == "Я"` (строгое сравнение строк).
   - **TRUE (ветка «Я»):** ⛔ **выход не подключён** (см. раздел 6, дефект №1). По замыслу должен идти в `Aggregate1 → Code8 → AI Agent2 → Code3 → «Карты Я»`.
   - **FALSE (ветка «МЫ»):** `Aggregate2 → Code7 → AI Agent3 → Code4 → «Карты МЫ»`.
4. Ветка **МЫ** (рабочая):
   - **Aggregate2** — собрать всё в `data` (исключая `Оценка`).
   - **Code7** — оставить только `{ Номер, Название_кейса (из «Название кейса»), Суть }`, завернуть в `{ кейсы: [...] }`.
   - **AI Agent3:** user=`JSON.stringify($json.кейсы)`; system=`Тема колоды`+`Аудитория`+**`Генерация карт МЫ`**+`Инструкция`. Модель `gemini-3.1-pro-preview`, json.
   - **Code4** — парсит JSON-массив, мапит в поля карты: `номер_карты (Карта_Id), исходный_кейс (Номер_кейса), название, луч (Луч), описание (Description), задача (Task), допматериал (Доп_материал), типЗадачи (Доп. материал)`.
   - **Append or update row in sheet1** → `Карты МЫ`, матчинг по `Номер`. Маппинг: Номер←номер_карты, Название, Описание, Задание←задача, Допматериал, Источник=`=` (пусто), Исходный кейс, Тип←**луч**.
5. Ветка **Я** (по замыслу, сейчас разорвана):
   - **Code8** — идентичен Code7 (`{ Номер, Название_кейса, Суть }`).
   - **AI Agent2:** system использует **`Генерация карт Я`** вместо «МЫ». Модель `gemini-3.1-pro-preview`, json.
   - **Code3** — парсит, мапит в `{ номер_карты (№ Порядковый номер карты/№), исходный_кейс, название, тип (ТИП/Луч), описание, задача, допматериал }`.
   - **Append or update row in sheet** → `Карты Я`, матчинг по `Номер`.

### Конвейер 4 — Медиа-промпты для карт «Я» (контур «МАЯК ТЕСТ»)
**Назначение:** для каждой готовой карты «Я» сгенерировать промпты для производства медиа (Текст/Аудио/Изображение/Видео/Интерактив/Данные).

1. **Google Sheets Trigger** — `Базовые промты` (МАЯК ТЕСТ).
2. **Get row(s) in sheet2** — читает `Карты Я` (МАЯК ТЕСТ).
3. **Aggregate3** — собрать в `data`, исключая `Промт, row_number, Исходный кейс, Допматериал, Источник`.
4. **AI Agent4:** user=`JSON.stringify($json.data)`; system=`{{ Карты я }}` (одна ячейка-промпт). Модель `gemini-3-pro-preview`, json.
5. **Code in JavaScript1** — разбирает текстовый вывод: режет по разделителю `\n───\n`, из каждого блока вытаскивает `№(\d+)` → number, второй абзац → prompt, чистит префикс типа (`Текст/Аудио/Изображение/Видео/Интерактив/Данные`).
6. **Append or update row in sheet2** → `Карты Я` (МАЯК ТЕСТ), матчинг по `Номер`, пишет `Номер`, `Промт`.

### Конвейер 5 — Медиа-промпты для карт «МЫ» (контур «МАЯК ТЕСТ»)

- **Code5** заметно сложнее Code1: обрабатывает 2 варианта (элемент-строка с Markdown — заголовок снаружи/внутри ``` ``` ```; элемент-объект с `card_id/header` либо `Номер/Промт`), чистит обёртки ` ```text/markdown/json `, добирает номер из начала текста.

### Конвейер 6 — Генерация доп. материалов в Google Docs
**Назначение:** для строк из листа создать форматированный справочный документ Google Docs.

1. **Google Sheets Trigger5** — `Законодательная власть` / `Лист9`.
2. **Loop Over Items** (splitInBatches) → выход «loop» → **Wait** (пауза/троттлинг).
3. **AI Agent6** — большой системный промпт «генератор доп. материалов» (полностью приведён в Приложении A). Ключевое: генерирует **входные материалы** (справки/таблицы/сценарии), НЕ решение; лимит 2000 символов; строгие правила Markdown; при упоминании реальной компании — веб-поиск. user=`Номер+Название+Описание+Задание`. Модель `gemini-3-flash-preview`, `responseFormat: text`.
4. **Create a document** (Google Docs) — в папке `1SVZwHCxAbxdRIJjog2n4LO62bSvavX1H`, заголовок `Справочный материал - {{ $now.format('dd.MM.yyyy HH:mm') }}`.
5. **Markdown** — конвертирует Markdown-вывод агента в HTML.
6. **Code in JavaScript6** — превращает HTML в массив запросов Google Docs API: вставка текста + `updateParagraphStyle` (HEADING_2/3) + `updateTextStyle` (bold). Использует маркеры `###H2###/###BOLD###` и пересчёт индексов.
7. **HTTP Request** — `POST https://docs.googleapis.com/v1/documents/{docId}:batchUpdate`, тело `{ requests }`, авторизация OAuth2 (genericCredentialType).

## 6. Обнаруженные дефекты и риски (важно для переноса)

1. **🔴 Ветка «Я» в конвейере 3 разорвана.** У ноды `If` выход TRUE (`ТИП=="Я"`) ни к чему не подключён, а `Aggregate1` не имеет входа. Значит **карты «Я» через этот конвейер сейчас не генерируются** — реально работает только ветка «МЫ». При переносе логику «Я» надо восстановить (If.true → Aggregate1).
2. **🟠 Loop без возврата.** `Loop Over Items` (splitInBatches) имеет выход «loop» → Wait → … → HTTP, но **обратной связи от HTTP к Loop нет**. Цикл не итерируется по всем строкам — обрабатывается фактически один батч/элемент. Для пакетной генерации Docs это баг.
3. **🟠 Хрупкий парсинг ответов ИИ.** Логика держится на `JSON.parse(output)` + регэкспы/«хирургия» битого JSON (Code2, Code1, Code5). Это симптом того, что модель не всегда отдаёт валидный JSON. При переносе — перейти на строгий structured output / JSON-schema и валидацию.
4. **🟠 Разнобой в ключах.** Код перебирает множество альтернативных названий полей (`№/Номер/number/card_id/id`, `Доп_материал/Доп. материал`). Признак нестабильного контракта между промптом и кодом. Нужна единая схема.
5. **🟡 Polling everyMinute по листу `Базовые промты`** в 4 триггерах одного документа → лишняя нагрузка и риск повторных запусков на одно изменение. На новой платформе — событийная модель/идемпотентность.
6. **🟡 Дубли кода.** Code7 ≡ Code8; Code4 ≡ Code9; Code3/Code4 почти идентичны. Кандидаты на единую функцию-маппер.
7. **🟡 Затирание полей.** `Update row in sheet` (П2) пишет `Тип проблемы = ""`; `Карты МЫ` пишет `Источник = ""`. Проверить, не теряются ли данные.
8. **🟡 Смешение контуров** «МАЯК Артек» (прод) и «МАЯК ТЕСТ» в одном активном workflow — риск записать тест в прод и наоборот.
9. **🟡 Секреты/доступы.** Завязка на Google OAuth2 (Sheets/Docs/Drive) и ключ OpenRouter — при переносе нужен аналог секрет-менеджмента.

## 7. Что усилить при переносе на платформу (рекомендации)

**Данные**

**ИИ-слой**

## 10. Методология «МАЯК» (LIGHTHOUSE) — содержательное ядро системы

Это **главный раздел** для пересборки: здесь не техника n8n, а смысл, который надо сохранить и усилить. Всё извлечено из листа `Базовые промты` (полные тексты — в `_promts_bazovye.txt`).

### 10.1. Общая рамка (конфигурация колоды)

Колода настраивается 4 «рамочными» ячейками + методологическим ядром:
- **Тема колоды:** «проект Московского долголетия».
- **Аудитория:** управленческая команда и профильные специалисты Московского долголетия.
- **Инструкция (формат вывода):** жёсткое требование к ИИ — только чистый JSON-массив, без ```, без пояснений, экранирование кавычек, всё в одну строку. (Именно отсюда растут «хирургические» парсеры в Code-нодах — модель регулярно нарушает это требование.)
- **Системный промт:** дублирующее требование строго JSON `{...}` без обёрток.

### 10.2. Методологическое ядро — три матрицы

Ядро (ячейка `Ядро`, 3454 симв.) задаёт **матрицу поиска кейсов** «Московское долголетие» — таблицу пересечений:

**5 функциональных блоков (строки, М1–М5):**
1. **М1: Вовлечение и коммуникации** (сегментация 55+, реклама/SMM, PR, работа с управами/поликлиниками, отток участников).
2. **М2: Методология и программы** (адаптация занятий, учебные программы, отбор преподавателей, требования к помещениям, травматизм/жалобы).
3. **М3: Операционное управление** (заявления/учёт, расписание/логистика, нормирование нагрузки, договоры с подрядчиками, срывы мероприятий).
4. **М4: HR и развитие персонала** (волонтёры, адаптация, найм/оценка 360°, вузы, текучка/выгорание).
5. **М5: Цифровизация и аналитика** (личный кабинет, мобильное приложение/боты, обучение CRM, интеграции ЕМИАС/«Соцзащита», сбои/утечки).

**5 кластеров стейкхолдеров (столбцы, С1–С5):** С1 Участники · С2 Программа · С3 Команда · С4 Партнёры/Площадки · С5 Эффективность/Риски.

Пересечение **М×С** (например М1×С2) = адрес ячейки, по которому генерируется один поисковый промпт. Это «мозг» системы: гарантирует покрытие всего поля задач.

### 10.3. Конвейер 1 (детально) — генерация поисковых промптов

Промпт `Создание Базовых промтов` (роль: **Мастер-Конструктор промтов**):
- Генерирует **ровно 12 промптов**, каждый привязан к уникальной ячейке матрицы (формат М×С).
- Стратегия покрытия: каждый из 5 блоков М представлен ≥2 раза, кластеры С варьируются, приоритет — острые пересечения.
- Каждый промпт — это готовое ТЗ на веб-поиск 4–5 реальных кейсов: роль эксперта-аналитика, география Москва, период 2–3 года, только по одной организации из темы.
- **Стоп-лист тем** (важно перенести как фильтр): нарушения закона РФ, политика/религия/острые соцконфликты, критика госинститутов, жертвы/катастрофы, скандалы, унижение профдостоинства, «боевая» лексика (бойцы/герои/победы).
- Формат вывода: markdown-таблица без заголовков (`Название кейса | Суть | Ссылка | Тип проблемы`).

> ⚠️ Сам **веб-поиск кейсов** по этим 12 промптам в n8n-workflow отсутствует — на выходе только промпты в лист `Промты для поиска`. Поиск, видимо, выполняется отдельно/вручную, а результат попадает в `Кейсы(не валидированные)` (715 строк). Это разрыв в автоматизации — кандидат №1 на достройку при переносе (см. вопрос 1).

### 10.4. Конвейер 2 (детально) — валидация и скоринг кейсов

Промпт `Валидация` (роль: **Главный Методолог и Аудитор**) — 3 этапа:
- **Этап 1. Дедупликация:** найти кейсы об одной ситуации, оставить один («АКТИВЕН»), прочим — «ДУБЛИКАТ» + оценка 0.
- **Этап 2. Скоринг 0–100** по 5 критериям:
  - Соответствие теме/аудитории (0–25),
  - Конкретика и детализация (0–25),
  - Драматургия и конфликт (0–20),
  - Практический потенциал (0–20),
  - Эмоциональная вовлекающая сила (0–10).
  - Стоп-лист тем → 0 баллов.
- **Этап 3. Типизация (ключевая развилка «Я»/«МЫ»):**
  - **Группа «Я» — тип «Инструментальная»:** потенциал под 6 типов контента (ТЕКСТ, ДАННЫЕ, АУДИО, СТАТИКА/изображение, ДИНАМИКА/видео, ИНТЕРАКТИВ). Нужно создать конкретный артефакт-контент.
  - **Группа «МЫ» — тип «Стратегическая»:** потенциал под 6 лучей ЗВЕЗДА. Нужно разработать системный документ (матрица, дорожная карта, концепция, стратегия).
- **Целевой порог колоды:** не менее **16 активных кейсов «Я» с оценкой 80+** и **16 активных «МЫ» 80+**.
- Формат ответа: `№ | Аналитика | Оценка | Группа`.

### 10.5. Конвейер 3 (детально) — генерация карт «Я» и «МЫ»

Это ядро продукта. Оба промпта (≈7000 симв. каждый) построены одинаково, различаются осью классификации.

**Карты «Я»** (роль: Методист-конструктор) — ось = **6 типов контента**, у каждого 6 паттернов с типом задачи (Генерация/Трансформация):
- **ТЕКСТ, АУДИО, ИЗОБРАЖЕНИЕ, ВИДЕО, ИНТЕРАКТИВ, ДАННЫЕ** — полный список из 36 паттернов приведён в `_promts_bazovye.txt`.
- Логика: из n кейсов сгенерировать **36 карт** (по 6 на тип, каждый паттерн ровно 1 раз). Один кейс — максимум в 3 типах, не более 1 раза в типе.

**Карты «МЫ»** (роль: Стратегический консультант) — ось = **6 лучей ЗВЕЗДА**, у каждого 6 паттернов:
1. **Знание и навыки** (матрица компетенций, ПДР, должностная, методичка, анализ трендов, стратегическая презентация),
2. **Внешнее взаимодействие** (концепция мероприятия, профилирование ЦА, предложение, анализ потребностей, удержание партнёра, контент-стратегия),
3. **Единая образовательная среда** (SWOT, инвентаризация ИТ, аудит цифрового актива, прототип суперприложения, бренд, анализ обратной связи),
4. **Защита данных** (сценарии реагирования, экспресс-аудит ИБ, обучающая платформа, анализ инцидентов, сравнение решений, compliance),
5. **Данные и аналитика** (предиктивный прогноз, инвентаризация источников, дашборд, прогноз нагрузок, объективизация выбора, анализ обращений),
6. **Автоматизация** (чат-бот, точки автоматизации, метрики, сценарии коммуникации, дорожная карта, RPA).
- Та же логика 36 карт.

**Общий «секрет качества» обоих промптов** (это надо сохранить при усилении!):
- **Тест релевантности / «попадание в боль»:** карта засчитана, только если у участника реакция «это про нас!», «чёрт, да это же мы», «у нас скоро так будет». Равнодушное «ну ок» = карта провалена.
- **Принцип автономности:** карта самодостаточна, участник не видит исходный кейс.
- **Шаблон карты:** `№ исходного кейса | № карты | Название (15–20 симв.) | ТИП/Луч | Description (300–350 симв.) | Task (180–220 симв.) | Доп. материал (Генерация/Трансформация)`.
- **Технические ограничения формата:** без конкретных дат (только относительные), без упоминания «ИИ/нейросеть» (прямые глаголы «Разработайте/Создайте»), без англицизмов (KPI/ROI/CRM → русские эквиваленты).
- Обязательная география + название организации + конфликт + дефицит времени в Description.

### 10.9. Что это даёт для усиления (конкретика поверх раздела 7)

Теперь, зная методологию, усиление можно сформулировать предметно:
- **Матрицу М×С вынести в данные** (не в текст промпта): таблица блоков/кластеров → конфиг. Тогда покрытие можно проверять программно (каждая ячейка использована, нет дублей М×С).
- **Достроить разрыв П1→кейсы:** добавить реальный шаг веб-поиска (search-API/агент с инструментом) между «промптами для поиска» и «сырыми кейсами».
- **Скоринг (П2) сделать детерминированным:** вернуть структурированные баллы по 5 критериям отдельными полями (а не только итог), чтобы порог «80+» и квоты «16/16» считались кодом, а не на доверии к модели.
- **Паттерн-аллокация (П3):** распределение «36 карт = 6×6 паттернов, кейс ≤3 раз» — это комбинаторная задача; её надёжнее решать кодом-планировщиком, а ИИ оставить только генерацию текста карты по заданному (паттерн, кейс).
- **Тест релевантности «боли»** формализовать в отдельный LLM-judge со шкалой, чтобы отбраковывать слабые карты автоматически.
- **Систему изображений LIGHTHOUSE** (фон-константа + гамма по типу) — это чистая шаблонизация, её вообще не нужно отдавать LLM целиком: гамма и фон подставляются кодом, ИИ генерирует только `[образ]` и `[настроение]`.
